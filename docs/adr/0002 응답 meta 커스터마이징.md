# 2. 응답 본문 meta 필드 기반 응답 커스터마이징

## Status
승인 (2026.06.19)

## Context
API의 개성을 보강하기 위해 응답에 부가 컨텍스트(예: 어느 도메인으로 처리됐는지)를 싣고 싶었다. 다만 이미 운영 중인 클라이언트가 있어 **기존 응답 스펙을 깨면 안 된다**는 제약이 있었다.

- 헤더에 싣는 방식은 클라이언트가 잘 읽지 않아 활용도가 낮다.
- 응답 본문에 필드를 추가하면 기존 클라이언트의 역직렬화에 영향을 줄 수 있다.

## Decision
응답 본문에 `meta`(`Map<String, Object>`) 필드를 두고, `ResponseBodyAdvice`로 직렬화 직전에 주입한다.

- `ResponseCustomizingAdvice`(`@RestControllerAdvice`)가 `ResponseCustomizer` 구현체들을 적용한다.
- `HostResponseCustomizer`는 요청 호스트가 `app.response.proxied-host` 설정값과 일치하면 `servedVia`를 넣는다.
- `meta`는 `@JsonInclude(NON_EMPTY)`라 비어 있으면 직렬화에서 제외된다 → 기존 응답은 그대로 유지(하위호환).

## Consequences
- `ApiResponse`를 반환하는 경로(clients 등)는 조건 충족 시 `meta`가 추가되고, 그 외에는 기존과 동일하게 직렬화된다.
- **남은 갭**: 적용 조건이 `body instanceof ApiResponse`인데, 메인 필터 엔드포인트 `/api/v1/filter`는 별도 record인 `FilterApiResponse`를 반환하므로 `meta`가 붙지 않는다. 트래픽 대부분을 차지하는 경로에 미반영 상태다.
- 통일하려면 `FilterApiResponse`에도 `meta`를 도입하거나 공통 응답 래핑을 재설계해야 한다 (후속 결정 필요).

## Alternatives
- 응답 헤더(X-Served-Via 등): 클라이언트 활용도 낮아 제외.
- 모든 응답을 단일 래퍼로 통일: 영향 범위가 크고 기존 스펙 파손 위험이 있어 이번 범위에서 제외.
